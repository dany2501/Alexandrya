RF-001: Registro de Usuarios

3.2 Contexto del negocio

flowchart LR
    V[Visitante] --> R[Formulario de registro]
    R --> C{¿Datos válidos?}
    C -->|No| E[Error de validación]
    C -->|Sí| N[Cuenta NO verificada]
    N --> M[Correo de verificación]
    M --> OK[Cuenta verificada]
    OK --> P[Puede pagar suscripción]
    R -. opcional .-> REF[Código de referido]

5.1 Flujo de registro

flowchart TD
    START([Visitante en registro]) --> A[Captura correo, contraseña, términos]
    A --> B{¿Anti-bot OK?}
    B -->|No| Z[❌ Rechazo silencioso]
    B -->|Sí| C{¿Formato y política OK?}
    C -->|No| D[❌ Error de validación] --> A
    C -->|Sí| E{¿Correo ya existe?}
    E -->|Sí| F[❌ Mensaje genérico anti-enumeración] --> A
    E -->|No| G[Crea cuenta NO verificada + hash BCrypt]
    G --> H{¿Código de referido?}
    H -->|Sí| I[Vincula referido pendiente]
    H -->|No| J[Continuar]
    I --> J
    J --> K[Genera token verificación 24h]
    K --> L[Encola correo NT-001]
    L --> M([✅ Mostrar verifica tu correo])

5.2 Secuencia

sequenceDiagram
    actor V as Visitante
    participant C as Cliente Web/App
    participant API as API
    participant DB as PostgreSQL
    participant MQ as RabbitMQ

    V->>C: completa formulario
    C->>API: POST /api/v1/auth/register
    API->>API: valida anti-bot, formato, política
    API->>DB: ¿existe email?
    DB-->>API: no existe
    API->>DB: INSERT usuario (no verificado, hash)
    opt código referido
        API->>DB: INSERT referido (estado=registrado)
    end
    API->>MQ: evento registro (NT-001)
    API-->>C: 201 Created
    C-->>V: "Verifica tu correo"

    TokenExpirado --> NoVerificada: reenviar verificación
    Verificada --> [*]

    end

    AUTH -->|válido| OK
    AUTH -->|fallo| NO

5.1 Flujo de inicio de sesión

flowchart TD
    START([Alumno en login]) --> A[Ingresa correo/contraseña]
    A --> B{¿Cuenta existe?}
    B -->|No| C[❌ Credenciales inválidas]
    C --> A
    B -->|Sí| D{¿Correo verificado?}
    D -->|No| E[❌ 403 Verifica tu correo] --> ENDV([Pantalla verificación])
    D -->|Sí| F{¿Rate limit OK?}
    F -->|No| G[❌ 429 Bloqueado 15 min] --> ENDB([Esperar])
    F -->|Sí| H{¿Contraseña correcta?}
    H -->|No| I[Intentos += 1 + auditar] --> A
    H -->|Sí| J{¿MFA activo?}
    J -->|Sí| K{¿2º factor válido?}
    K -->|No| I
    K -->|Sí| L[Invalida sesión previa]
    J -->|No| L
    L --> M[Crea sesión única + tokens]
    M --> N[Registra acceso + correo aviso]
    N --> O([✅ Sesión iniciada])

5.2 Diagrama de secuencia

sequenceDiagram
    actor U as Alumno
    participant C as Cliente (Web/App)
    participant API as API (NestJS)
    participant DB as PostgreSQL
    participant R as Redis
    participant MQ as RabbitMQ

    U->>C: correo + contraseña
    C->>API: POST /api/v1/auth/login
    API->>DB: SELECT usuario WHERE email
    DB-->>API: usuario (hash, verificado, estado)
    alt no verificado
        API-->>C: 403 Email no verificado
    else verificado
        API->>R: ¿rate limit IP?
        API->>API: bcrypt.compare(password, hash)
        opt MFA activo
            API-->>C: 401 mfa_required
            U->>C: código MFA
            C->>API: POST /auth/login (mfa_code)
        end
        API->>R: invalida sesión previa + set sesión activa
        API->>DB: INSERT sesion + acceso_historial
        API->>MQ: evento nuevo_acceso (NT-002)
        API-->>C: 200 {access_token, refresh_token, usuario}
        C-->>U: dashboard
    end

5.3 Estados de la sesión

stateDiagram-v2
    [*] --> Validando: envía credenciales
    Validando --> Fallida: inválidas
    Fallida --> Bloqueada: 5 intentos/15min
    Bloqueada --> Validando: pasados 15 min
    Validando --> Activa: ✓ credenciales (+MFA)
    Activa --> Activa: usa access token
    Activa --> Expirada: pasa 1 h
    Expirada --> Activa: refresh válido (< 30d)
    Expirada --> Reautenticar: refresh expirado
    Activa --> Invalidada: login en otro dispositivo
    Activa --> Cerrada: logout
    Invalidada --> [*]
    Cerrada --> [*]
    Reautenticar --> [*]

5.1 Flujo de recuperación

flowchart TD
    START([Olvidé mi contraseña]) --> A[Ingresa correo]
    A --> B[Respuesta genérica siempre]
    B --> C{¿Correo existe y verificado?}
    C -->|No| Z[No envía nada]
    C -->|Sí| D[Genera token reset 1h + invalida tokens previos]
    D --> E[Correo NT-004 con enlace]
    E --> F[Usuario abre enlace]
    F --> G{¿Token válido y vigente?}
    G -->|No| H[❌ 410 Solicita uno nuevo]
    G -->|Sí| I[Define nueva contraseña + política]
    I --> J[Actualiza hash + marca token usado]
    J --> K[Invalida TODAS las sesiones]
    K --> L[Correo NT-003 + a login]
    L --> END([✅ Contraseña restablecida])

5.2 Secuencia de cambio autenticado

sequenceDiagram
    actor A as Alumno
    participant API as API
    participant DB as PostgreSQL
    participant R as Redis
    participant MQ as RabbitMQ

    A->>API: POST /auth/password/change (actual, nueva) + Bearer
    API->>DB: verifica contraseña actual (BCrypt)
    alt actual incorrecta
        API-->>A: 401 contraseña_actual_invalida
    else correcta
        API->>DB: UPDATE password_hash
        API->>DB: invalida todas las sesiones
        API->>R: purga sesiones en caché
        API->>MQ: evento cambio_contraseña (NT-003)
        API-->>A: 200 (debe reautenticar)
    end

    RF001B --> RF080[RF-080 Sesión única]

    RF001B --> DB[(usuarios / tokens_reset / sesiones)]

Diagrama — Flujos de negocio

09-diagramas/04-flujos.md

1. Registro + pago + activación de suscripción

sequenceDiagram
    actor V as Visitante
    participant W as Web (Next.js)
    participant API as API (NestJS)
    participant PAY as Pasarela de pago
    participant MQ as RabbitMQ
    participant MAIL as Correo

    V->>W: Registro (correo + contraseña)
    W->>API: POST /auth/register
    API->>MQ: evento "registro"
    MQ->>MAIL: Correo de verificación
    V->>API: GET /auth/verify (enlace)
    API-->>V: Cuenta verificada

    V->>W: Elige plan y paga
    W->>API: POST /checkout
    API->>PAY: Crea sesión de pago
    PAY-->>V: Flujo de pago (tarjeta/SPEI)
    PAY-->>API: Webhook pago confirmado
    API->>API: Idempotencia (RN-021) + activa suscripción 365d (RN-011)
    API->>MQ: evento "pago_exitoso"
    MQ->>MAIL: Correo de pago exitoso
    API-->>W: Suscripción activa

2. Login con control de sesión única

sequenceDiagram
    actor AL as Alumno
    participant CLI as Cliente (Web o App)
    participant API as API
    participant R as Redis
    participant MQ as RabbitMQ
    participant MAIL as Correo

    AL->>CLI: Inicia sesión
    CLI->>API: POST /auth/login (+ MFA si aplica)
    API->>R: ¿Existe sesión activa previa?
    alt Hay sesión previa
        API->>R: Invalida sesión anterior (RN-031)
    end
    API->>R: Registra sesión actual como única
    API->>API: Emite JWT + refresh token
    API->>MQ: evento "nuevo_acceso" (IP, dispositivo)
    MQ->>MAIL: Correo de aviso de acceso (RN-034)
    API-->>CLI: Tokens + sesión activa
    Note over CLI,API: El dispositivo anterior queda expulsado al usar su token

3. Realizar una evaluación

sequenceDiagram
    actor AL as Alumno
    participant CLI as Cliente
    participant API as API
    participant DB as PostgreSQL

    AL->>CLI: Selecciona evaluación (tema/módulo/.../simulador)
    CLI->>API: GET /evaluaciones/{config}
    API->>API: Valida suscripción activa (RN-010)
    API->>DB: Arma preguntas según config (RN-050)
    API-->>CLI: Set de preguntas (sin respuesta correcta)
    loop Por cada pregunta
        AL->>CLI: Responde opción
    end
    CLI->>API: POST /intentos (respuestas)
    API->>API: Califica (RN-054) y, si simulador, valida tiempo (RN-051)
    API->>DB: Guarda intento + respuestas (RF-044)
    API-->>CLI: Calificación + retroalimentación + explicaciones (RF-043)

4. Estados de la suscripción

stateDiagram-v2
    [*] --> SinSuscripcion
    SinSuscripcion --> Activa: Pago confirmado (RN-020)
    Activa --> Activa: Renovación anticipada (extiende, RN-013)
    Activa --> Vencida: Llega fin_vigencia (RN-012)
    Vencida --> Activa: Renovación tardía (nueva vigencia 365d, RN-014)
    Activa --> Cancelada: Cancelación / reembolso
    Cancelada --> Activa: Nuevo pago confirmado
    Vencida --> [*]
    Cancelada --> [*]
    note right of Vencida
        Cuenta y historial se conservan (RN-015);
        solo se bloquea el acceso al contenido.
    end note

stateDiagram-v2

    Pendiente --> Fallido: Rechazo / timeout
    Confirmado --> Reembolsado: Reembolso
    Fallido --> [*]
    Reembolsado --> [*]

6. Referido efectivo y otorgamiento de beneficio

sequenceDiagram
    actor A as Alumno (refiere)
    actor B as Invitado
    participant API as API
    participant DB as PostgreSQL

    A->>API: Comparte código de referido (RN-045)
    B->>API: Se registra con el código
    API->>API: ¿Auto-referido? (RN-044) -> rechaza si aplica
    API->>DB: Crea REFERIDO estado=registrado
    B->>API: Paga su suscripción
    API->>API: ¿Posición <= 3? (RN-040)
    alt Dentro del límite
        API->>DB: REFERIDO -> efectivo (RN-041)
        API->>DB: Otorga beneficio configurado, idempotente (RN-042, RN-043)
    else Excede el límite
        API->>API: No otorga beneficio adicional
    end
